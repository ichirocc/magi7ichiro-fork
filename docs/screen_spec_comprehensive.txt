 PART A — 設計原則と恒久決定
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ UI原則
 1. 状態駆動の単一CTA   状態(未作成/実行中/もう少し/配れます)に応じ大ボタンが1つ自動変化
 2. 一本指・ドラッグ不可  全操作はタップのみ。ドラッグ/ピンチに依存しない
 3. 専門用語を出さない   一般画面はcovU/c3/ALNS等を出さず「人手不足/並び/直す」等の平易語
 4. 違反は三重符号化     色＋形(実線/破線)＋集計ハイライト＝色覚・印刷に頑健
 5. 破壊操作は安全弁     上書き系は確認ダイアログ(既定キャンセル)＋Undo/Redo
 6. 誤操作の構造防止     担当不可シフトは選択肢に出さない／依存未充足は誘導文
 7. 重複は意味で判断     同一データの多重表示は集約。ただし編集文脈で要る情報は残す
 8. 1指標1名称         同じ「良さ」を別名・別単位で出さない
 9. プロ機能は隔離       生指標・実験フラグはプロ表示/詳細設定に隔離

■ 恒久決定（再提案しない・業務確定）
 P1 use2 は常時ON（将来も恒久）。必要数2は常に有効・必須級・常時表示。
    被覆は常に need1/need2 の OR（ゆるい方で充足）。
 P2 C41 / C42 / C41s / C42s は 2×2 の4独立ファミリー（群/スキル × 回数/組合せ）。
    集約しない。採点も別キー。
 HF77 規律: パラメータ/重み/データの変更は業務オペレータの明示的 numeric 指示が必要。
    一度に1変更、A/B検証必須。
 多実装一貫性: VBA版・Web版・Android版で用語/モデルを揃える（例「グループ」は3版共通。
    Android単独の改名は不可）。
 ws8/ws9新設・Cells→CodeName移行・Mod6新設・各種自動検出 は不要（恒久）。

■ 採点モデル（MirrorKeys.weights = 単一の真実。最適化器と表示が同じ重みを参照）
 HARD（絶対）: グループ不整合10000 ＞ 希望違反9000 ＞ 人員不足8000 ＞ 禁止の並び7000
 SOFT（できれば): 下限割れ90 ＞ 上限超過45 ＞ 回避の並び12 ＞ 窓の要件4 ＞ 必須の並び3
              ＞ 推奨の並び2 ＞ 個人合計/群レンジ/群ペア/スキルレンジ/スキルペア/適切回数/
                公平化=各1 ＞ 過剰な配置0.5
 ・できあがり度 = 初期からの違反削減率(0-100)。配布可否は HARD=0 で判定。

■ 制約ファミリー全体像
 cons1=期間の窓(c1) / cons2=個人合計(c2)
 cons3=必須の並び(HARD) / cons3n=禁止の並び(HARD) / cons3m=推奨(SOFT) / cons3mn=回避(SOFT)
 cons41=群×シフト回数(C41) / cons42=群ペア禁止(C42)
 cons41s=スキル×シフト回数(C41s) / cons42s=スキルペア禁止(C42s)
 apt=適切回数(群1人あたり目標・双方向L1) / fair=群内公平化(L1)
 low/high=個人別下限/上限 / covU/covO=人員不足/過剰 / pref=希望 / groupViol=群整合
