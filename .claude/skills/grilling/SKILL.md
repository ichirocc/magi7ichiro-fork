# grilling（要件の壁打ち）

作る前に、この計画・設計のあらゆる面について**容赦なく質問**し、認識が完全に一致するまで詰める。設計ツリーの各分岐を一つずつ辿り、決定どうしの依存関係を順番に解消していく。**各質問には必ず「自分が推奨する答え」と一行の根拠を添える**（相手が即答できる形にする）。

## 進め方（鉄則）
- **質問は一度に一つだけ**。複数同時はNG（混乱する）。前の回答を得てから次の質問へ進む。
- コードや既存ドキュメントで答えが分かることは**質問せず自分で調べる**（`docs/architecture.md`・`docs/screen_spec.md`・`docs/magi_design_system.md`・該当ソース）。「分からないこと」だけを聞く。
- 回答はこの利用者の流儀に合わせる：**簡潔・日本語・結論先出し**。
- モバイル運用が多いので、選択肢があるときは**タップで選べる形（ask_user_input）**で出す。自由記述が要るものだけ文章で聞く。
- 認識が一致したら詰めを終了し、**決定事項を3〜5行で要約**してから実装に入る。

## MAGIで必ず詰める観点
1. **対象実装**：Android(ネイティブ) / Web(`magi_v6_web.html`) / VBA のどれか。複数なら同期方針も。
2. **変更の種類**：①ロジック ②表示・ドキュメント ③値チューニング のどれか。
   - ③値チューニング（重み・パラメータ・データ値）は **HF77** に従い、**明示的な数値指示が無い限り勝手に変えない**。grilling で詰めるのは「意図」であって自動調整ではない。
3. **不変条件（Level Zero）に触れるか**：ws1–ws7 レイアウト／`RW_N1=46`／`RW_N2=56`／モジュール構成（Mod1–Mod5）などを壊さないか。壊すなら止めて確認。
4. **制約系の特定**：c1(窓)／c2(個人合計)／c3系(連勤パターン: MUST/禁止/Want/Hate)／c41・c42(群)／被覆(covU/covO)／希望(ws3 = `wishes`)。**ws3 と cons3系を混同しない**。
5. **完了条件**：何をもって「できた」とするか（HARD=0？ score のしきい値？ 実機確認？ CIビルド通過？）。

## いつ使うか
- 新機能・リファクタ・仕様判断・移植など、**非自明な作業に着手する前**。
- 「grill」「詰める」「壁打ち」「要件を固めたい」などの合図があったとき。
- 逆に、答えが一つに決まっている瑣末な作業や、利用者が既に詳細な指示を出している場合は使わない（過剰な質問は逆効果）。

## 出典
Matt Pocock の `mattpocock/skills`（productivity/grilling）を MAGI 向けに翻案。原典の中核「容赦なく一問ずつ・各問に推奨案を添える・コードで分かることは調べる」を踏襲しつつ、MAGI の三実装・HF77・Level Zero・制約系の観点を追加した。
